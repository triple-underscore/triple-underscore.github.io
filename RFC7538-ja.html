HTTP Status Code 308 (Permanent Redirect)

RFC 7538

要約

この文書は、追加の~HTTP状態code `308$st を指定する。 ◎ This document specifies the additional Hypertext Transfer Protocol (HTTP) status code 308 (Permanent Redirect).

1. 序論

~HTTPは、要請を異なる~URI（ `RFC3986$r ）へ~redirectする目的~用に一式の`状態code$を定義する。 これらの状態codeの歴史は `7231-6.4$rfc に要約されており，既存の状態codeを 4 種に分類している。 ◎ HTTP defines a set of status codes for the purpose of redirecting a request to a different URI ([RFC3986]). The history of these status codes is summarized in Section 6.4 of [RFC7231], which also classifies the existing status codes into four categories.

これらには、まず，下に分類できるような状態code［ `301$st, `302$st, `307$st ］がある： ◎ The first of these categories contains the status codes 301 (Moved Permanently), 302 (Found), and 307 (Temporary Redirect), which can be classified as below:

◎ 恒久的 ◎ Permanent 一時的 ◎ Temporary
要請~methodを `POST$m から `GET$m へ変更することを許容する ◎ Allows changing the request method from POST to GET `301^st0 `302^st0
要請~methodを `POST$m から `GET$m へ変更することを許容しない ◎ Does not allow changing the request method from POST to GET — `307^st0

`7231-6.4.7$rfc では、状態code `307$st0 に対する恒久的な変種を定義しないものと言明している。 この仕様は、この欠落している変種を定義する状態code `308$st0 を追加する。 ◎ Section 6.4.7 of [RFC7231] states that it does not define a permanent variant of status code 307; this specification adds the status code 308, defining this missing variant (Section 3).

この仕様には、この仕様が廃用する試験的（ `Experimental^en ） RFC 7238 からの技術的な変更点はない。 ◎ This specification contains no technical changes from the Experimental RFC 7238, which it obsoletes.

3. `308^st0 恒久的~redirect

`状態code$ `308^st は、［ `~target資源$には，新たな恒久的~URIが割当てられていて、この資源への未来の参照は，同封された いずれかの~URIを利用する~OUGHT ］ことを指示する。 ~link編集~能力を備えている`~client$は、可能0な所では，［ `実効~要請~URI$への参照 ］を［ `~server$から送信されてきた，一つ以上の新たな参照 ］に，自動的に~linkし直す~OUGHT。 ◎ The 308 (Permanent Redirect) status code indicates that the target resource has been assigned a new permanent URI and any future references to this resource ought to use one of the enclosed URIs. Clients with link editing capabilities ought to automatically re-link references to the effective request URI (Section 5.5 of [RFC7230]) to one or more of the new references sent by the server, where possible.

`~server$は、応答~内に［ 新たな恒久的~URI用に選好される~URI参照を包含している `Location$h ~header ］を生成する~SHOULD。 `~UA$は、 `Location^h ~header値を自動的~redirection用に利用しても~MAY。 ~serverの応答`~payload$は、通例的に新たな~URI（たち）への~hyperlinkを伴う短い~hypertextを包含する。 ◎ The server SHOULD generate a Location header field ([RFC7231], Section 7.1.2) in the response containing a preferred URI reference for the new permanent URI. The user agent MAY use the Location field value for automatic redirection. The server's response payload usually contains a short hypertext note with a hyperlink to the new URI(s).

`308^st0 応答は、`既定で~cache可能である$。 ◎ A 308 response is cacheable by default; i.e., unless otherwise indicated by the method definition or explicit cache controls (see [RFC7234], Section 4.2.2).

注記： この状態codeは、 `301$st に類似する — 要請~methodを `POST$m から `GET$m へ変更することは許容しないことを除き。 ◎ Note: This status code is similar to 301 (Moved Permanently) ([RFC7231], Section 6.4.2), except that it does not allow changing the request method from POST to GET.

4. 配備~上の考慮点

`7231-6$rfc では、［ 受信者は、未知の `3xx$st0 状態codeを `300$st と同じ仕方で扱う ］ものと要求する。 したがって，~serverは、状態code［ `301$st0, `302$st0, `307$st0 ］と同様に，自動的~redirectionが起こることには依拠-可能でないことになる。 ◎ Section 6 of [RFC7231] requires recipients to treat unknown 3xx status codes the same way as status code 300 (Multiple Choices) ([RFC7231], Section 6.4.1). Thus, servers will not be able to rely on automatic redirection happening similar to status codes 301, 302, or 307.

したがって，状態code `308^st0 の利用は、~serverが［ ~clientが新たな~codeを解するか，状態code `300^st0 の意味論へ~fallbackしても問題にならない ］と~~確信できる事例に制約される。 ~serverの実装者には、状態codeを要請の特性 — `User-Agent$h ~headerなど（ “`User-Agent Sniffing^en” ） — に基づいて変えないことを勧める。 そうすると、通例的に~codeを保守するのも~debugするのも難しくなることに加え，~cachingにも特別な注意を（すなわち、 `Vary$h 応答~headerの設定を）要するので。 ◎ Therefore, the use of status code 308 is restricted to cases where the server has sufficient confidence in the client's understanding the new code or when a fallback to the semantics of status code 300 is not problematic. Server implementers are advised not to vary the status code based on characteristics of the request, such as the User-Agent header field ("User-Agent Sniffing") — doing so usually results in code that is both hard to maintain and hard to debug and would also require special attention to caching (i.e., setting a "Vary" response header field, as defined in Section 7.1.4 of [RFC7231]).

~HTMLに基づく既存の~UAの多くは、~HTMLの `<meta http-equiv=refresh>^c 指令に遭遇したときに，~refreshを模倣することに注意。 これは、別の~fallbackとしても利用できる — 例えば： ◎ Note that many existing HTML-based user agents will emulate a refresh when encountering an HTML <meta> refresh directive ([HTML], Section 4.2.5.3). This can be used as another fallback. For example:

~client要請： ◎ Client request: 

GET / HTTP/1.1
Host: example.com

~serverからの応答： ◎ Server response: 

HTTP/1.1 308 Permanent Redirect
Content-Type: text/html; charset=UTF-8
Location: http://example.com/new
Content-Length: 356

<!DOCTYPE HTML>
<html>
   <head>
      <title>Permanent Redirect</title>
      <meta http-equiv="refresh" 
            content="0; url=http://example.com/new">
   </head>
   <body>
      <p>
         The document has been moved to
         <a href="http://example.com/new"
         >http://example.com/new</a>.
      </p>
   </body>
</html>

5. 保安上の考慮点

~HTTP~redirectに適用されるすべての保安上の考慮点は、状態code `308$st0 にも適用される（ `7231-9$rfc を見よ）。 ◎ All security considerations that apply to HTTP redirects apply to the 308 status code as well (see Section 9 of [RFC7231]).

~Internet越しの保安的でない通信は、中間者による~messageの改変の対象になる — 状態codeや~redirect~targetを変更することを含めて。 それらの攻撃を軽減する仕方として TLS（ Transport Layer Security ）の利用が挙げられる。 権限と~message完全性に関係する攻撃については、 `7230-9$rfc を見よ。 ◎ Unsecured communication over the Internet is subject to man-in-the-middle modification of messages, including changing status codes or redirect targets. Use of Transport Layer Security (TLS) is one way to mitigate those attacks. See Section 9 of [RFC7230] for related attacks on authority and message integrity.
