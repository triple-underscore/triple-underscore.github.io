RFC 9875 — HTTP Cache Groups

◎要約

この仕様は、 ~HTTP~cache内に格納-済みな応答どうしの関係性を述べる手段を導入する — それは、 各~格納-済み応答に 1 個以上の文字列を結付けることにより，それらを~group化する。 ◎ This specification introduces a means of describing the relationships between stored responses in HTTP caches, grouping them by associating a stored response with one or more strings.

1. 序論

`HTTP-CACHING$r は、 単独の資源の粒度で演算する — 各~格納-済み応答の`鮮度$は、 他の格納-済み応答には影響しない。 この粒度は、 ~cachingを より効率的にし得る — 例えば、 ある~pageが［ ~cachingに対し異なる要件を課す多数の~asset ］から構成されるとき。 ◎ HTTP caching [HTTP-CACHING] operates at the granularity of a single resource; the freshness of one stored response does not affect that of others. This granularity can make caching more efficient -- for example, when a page is composed of many assets that have different requirements for caching.

しかしながら、 格納-済み応答どうしの関係性を~cache効率を改善するために利用できる事例もある。 ◎ However, there are also cases where the relationship between stored responses could be used to improve cache efficiency.

例えば、 一群の互いに関係する資源たちを無効~化することが必要yあることは多い。 例えば、 ある状態変更~要請が他の資源に対する副作用を伴うときや， 純粋に管理上の便利（例： “当の~siteを成す この部分を無効~化する” ）のために。 ~URL構造の様なものに依拠する代わりに， 応答たちを一緒に~group化することは、 これらの関係性を表出する専用な仕方を供する。 ◎ For example, it is often necessary to invalidate a set of related resources. This might be because a state-changing request has side effects on other resources, or it might be purely for administrative convenience (e.g., "invalidate this part of the site"). Grouping responses together provides a dedicated way to express these relationships, instead of relying on things like URL structure.

~group化により指示される関係性は、 無効~化~eventを共有することに加えて， ~cacheが自身の演算を最適化するためにも利用できる （例：~cache抹消~algoの演算を伝えるため【？】）。 ◎ In addition to sharing invalidation events, the relationships indicated by grouping can also be used by caches to optimise their operation (e.g., to inform the operation of cache eviction algorithms).

`Cache-Groups§h は、 ~HTTP~cache内に格納-済みな応答どうしの関係性を述べる手段を導入する — この関係性は、 それら各~応答を 1 個以上の~groupに結付けることにより反映される。 また、 各~cacheが［ ある~groupを成す~memberたちに無効~化~eventを適用する ］ために［ この関係性~情報を どう利用できるか ］も述べる。 ◎ Section 2 introduces a means of describing the relationships between stored responses in HTTP caches, by associating those responses with one or more groups that reflect those relationships. It also describes how caches can use that information to apply invalidation events to members of a group.

`Cache-Group-Invalidation§h は、 そのような無効~化~eventを成す~sourceとして，新たな~HTTP応答~headerを導入する — それは、［ ある~groupの無効~化を誘発するための状態変更~応答 ］を許容する。 ◎ Section 3 introduces one new source of such events: an HTTP response header field that allows a state-changing response to trigger a group invalidation.

これらの仕組みは、［ 単独の~cacheの中で，単独の`生成元~server$に結付けられた格納-済み応答たち ］にまたがって演算する （`~group化された応答の識別-法＠#identify§を見よ）。 これらの仕組みは、［ 複数の~cache （例：ある［ 階層／網 ］内にあるもの） どうしで状態を同期する課題に取組む ］ことも［ 別々な生成元からの格納-済み応答どうしの結付けを手助けする ］こともない。 ◎ These mechanisms operate within a single cache, across the stored responses associated with a single origin server (see Section 2.1). They do not address the issues of synchronising state between multiple caches (e.g., in a hierarchy or mesh), nor do they facilitate association of stored responses from disparate origins.

2. `Cache-Groups^h 応答~header

`Cache-Groups^h 応答~headerは、 `~sf~list$ `STRUCTURED-FIELDS$r を値にとる。 この~listを成す各~memberは、 `~sf文字列$を値にとり，当の応答が属する~groupを識別する。 これらの文字列は、 それを作成した~serverには何らかの意味があるかもしれないとしても， 不透明である — ~cacheは、 それらの［ 構造／内容 ］に関する洞察を何ら得られない （ある~groupを一意に識別すること以外には）。 ◎ The Cache-Groups response header field is a List of Strings (Sections 3.1 and 3.3.3 of [STRUCTURED-FIELDS]). Each member of the List is a value that identifies a group that the response belongs to. These Strings are opaque -- while they might have some meaning to the server that creates them, the cache does not have any insight into their structure or content (beyond uniquely identifying a group). 

HTTP/1.1 200 OK
Content-Type: application/javascript
Cache-Control: max-age=3600
Cache-Groups: "scripts"

~memberたちの順序付けは有意でない。 認識されない`~sf~parameter$は無視される。 ◎ The ordering of members is not significant. Unrecognised Parameters are to be ignored.

実装は、 少なくとも［ `~field値$において 32 個の~group, 各~memberは 32 文字 ］は~supportしなければナラナイ。 実施においては、 ~HTTP~field長さに対する汎用な制限が， この`~field値$の~sizeを拘束し得ることに注意。 ◎ Implementations MUST support at least 32 groups in a field value, with up to at least 32 characters in each member. Note that generic limitations on HTTP field lengths may constrain the size of this field value in practice.

2.1. ~group化された応答の識別-法

同じ~cache内に格納-済みな 2 個の応答は、 ~AND↓ を満たすとき，同じ~groupに属するものと見なされる 【~groupを “共有する” とも称される】： ◎ Two responses stored in the same cache are considered to belong to the same group when all of the following conditions are met:

  • どちらも `Cache-Groups^h 応答~headerを包含していて， それらの値は同じ`~sf文字列$を包含する（文字大小区別）。 ◎ They both contain a Cache-Groups response header field that contains the same String (in any position in the List), when compared character-by-character (case sensitive).
  • どちらも同じ~URI生成元を共有する （ `HTTP$r `~URI生成元＠~HTTPinfra#origin§に則って）。 ◎ They both share the same URI origin (per Section 4.3.1 of [HTTP]).

2.2. ~cacheの挙動

2.2.1. 無効~化

~cacheは、 ある格納-済み応答を無効~化するときは，［ 自身に格納-済みな応答のうち，当の応答と~groupを （`~group化された応答の識別-法＠#identify§に則って） 共有するもの ］を無効~化してもヨイ。 ~group化された無効~化により， ~group化された無効~化が更に誘発されることはないことに注意 — すなわち、 この仕組みは~~連鎖しない。 ◎ A cache that invalidates a stored response MAY invalidate any stored responses that share groups (per Section 2.1) with that response. Note that further grouped invalidations are not triggered by a grouped invalidation; i.e., this mechanism does not cascade.

~cache拡張は、 上の要件を明示的に強化できる。 例えば，`~targeted~cache制御~header$ `TARGETED$r は、 それを処理している~cacheに対し，［ そのような応答を無効~化することが要求される ］ものと指定するかもしれない。 ◎ Cache extensions can explicitly strengthen the requirement above. For example, a targeted cache control header field [TARGETED] might specify that caches processing it are required to invalidate such responses.

3. `Cache-Group-Invalidation^h 応答~header

`Cache-Group-Invalidation^h 応答~headerは、 `~sf~list$ `STRUCTURED-FIELDS$r を値にとる。 この~listを成す各~memberは、 `~sf文字列$を値にとり，当の応答が属する~groupを識別する （`~group化された応答の識別-法＠#identify§【！`無効~化＠#invalidation§】に則って）。 ◎ The Cache-Group-Invalidation response header field is a List of Strings (Sections 3.1 and 3.3.3 of [STRUCTURED-FIELDS]). Each member of the List is a value that identifies a group that the response invalidates, per Section 2.2.1.

例えば， 2 個の~cache~groupに対する副作用がある `POST$m 要請 【より一般には，`安全$でない~methodを伴う要請】 に対する応答は、［ それら［ いずれかの~group／両~group ］に属する格納-済み応答を無効~化するべきである ］ことも指示できる： ◎ For example, following a POST request that has side effects on two cache groups, the corresponding response could indicate that stored responses associated with either or both of those groups should be invalidated with: 

HTTP/1.1 200 OK
Content-Type: text/html
Cache-Group-Invalidation: "eurovision-results", "australia"

~cacheは、 所与の応答において `Cache-Group-Invalidation^h ~headerを受信したときは： ◎ ↓

  • 当の応答が応対した要請が`安全$な~method（例： `GET$m ）を伴う場合には、 当の~headerを無視しなければナラナイ。 ◎ The Cache-Group-Invalidation header field MUST be ignored on responses to requests that have a safe method (e.g., GET; see Section 9.2.1 of [HTTP]).
  • 他の場合、 自身に格納-済みな応答のうち， 当の~headerに~listされた いずれかの~groupを （`~group化された応答の識別-法＠#identify§に則って） 共有するものを無効~化してもヨイ。 ◎ A cache that receives a Cache-Group-Invalidation header field on a response to an unsafe request MAY invalidate any stored responses that share groups (per Section 2.1) with any of the listed groups.

~cache拡張は、 この要件を明示的に強化し得る。 例えば，`~targeted~cache制御~header$ `TARGETED$r は、 それを処理している~cacheに対し，［ `Cache-Group-Invalidation^h 通達を尊重することが要求される ］ものと指定するかもしれない。 ◎ Cache extensions can explicitly strengthen the requirement above. For example, a targeted cache control header field [TARGETED] might specify that caches processing it are required to respect the Cache-Group-Invalidation signal.

~memberたちの順序付けは有意でない。 認識されない`~sf~parameter$は無視される。 ◎ The ordering of members is not significant. Unrecognised Parameters are to be ignored.

実装は、 少なくとも［ `~field値$において 32 個の~group, 各~memberは 32 文字 ］は~supportしなければナラナイ。 実施においては、 ~HTTP~field長さに対する汎用な制限が， この`~field値$の~sizeを拘束し得ることに注意。 ◎ Implementations MUST support at least 32 groups in a field value, with up to at least 32 characters in each member. 5. ~securityの考慮点

この仕組みは、 ある生成元を共有する資源どうしが互いを無効~化することを許容する。 このことから、 複数の主体を表現する生成元（ “共用~host法（ `shared hosting^en ）” と称されることもある）は， ある主体が他の主体の資源たちに対し［ 自身の資源と~group化すること／ 副作用を伴う通達を送信すること ］を許容するかもしれない。 ◎ This mechanism allows resources that share an origin to invalidate each other. Because of this, origins that represent multiple parties (sometimes referred to as "shared hosting") might allow one party to group its resources with those of others or to send signals that have side effects upon them.

これらの~riskを軽減するよう望む共用~hostは、 この仕様において定義される~headerへの~accessを制御できる。 ◎ Shared hosts that wish to mitigate these risks can control access to the header fields defined in this specification.
