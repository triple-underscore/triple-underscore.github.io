2. 考慮する質問

2.1. この特能は、情報として何を~web~siteその他の主体に公開し得るか？ その公開が必要とされる目的は何か？

情報を~webに公開できるからといって、そうするベキであることにはならない。 その情報を ある生成元に公開することは ⇒＃ 利用者にどんな便益があるのか？ その便益は、潜在的な~riskに勝るほどのものか？ そうならば、どう勝るか？ ◎ Just because information can be exposed to the web doesn’t mean that it should be. How does exposing this information to an origin benefit a user? Is the benefit outweighed by the potential risks? If so, how?

この質問に回答することは、当の［ 特能／仕様 ］が可能化する利用事例を仕様の中で明瞭にして，それらの特能と~privacyとの~tradeoffを［ ~W3C~TAG（ `Technical Architecture Group^en ）／ ~W3C~PING（ `Privacy Interest Group^en ） ］が解する助けになることが多い。 ◎ In answering this question, it often helps to ensure that the use cases your feature and specification is enable are made clear in the specification itself to ensure that TAG and PING understand the feature-privacy tradeoffs being made.

2.2. この仕様は、特能が力を得る必要最小限な情報を公開しているか？

何が公開されるかを問わず、仕様は，欲される利用事例を達成するために必要とされる最小限を超える~dataを公開しているか？ そうならば、それはなぜで，なぜ追加的な情報を公開するのか？ ◎ Regardless of what data is being exposed, is the specification exposing the bare minimum necessary to achieve the desired use cases? If not, why not and why expose the additional information?

2.3. この仕様は、個人的~情報, 個人識別可能な情報, それらから導出される情報をどう処するか？

個人的~情報（ `personal informatin^en ）とは、［ 利用者についての~data（例：住所）／ 利用者を識別するためにも利用し得る情報（別名や~email~address） ］である。 これは、個人識別可能な情報（ ~PII — `personally identifiable information^en ）とは別物である — 何が~PIIと見なされるかの正確な定義は、【司法の】管轄ごとに変わるので。 ◎ Personally information is data about a user (home address) or information that could be used to identify a user (alias or email address). This is distinct from personally identifiable information (PII), as the exact definition of what’s considered PII varies from jurisdiction to jurisdiction.

考慮~下にある仕様が［ 個人（ `individual^en ）を識別することもできるような［ 個人的~情報, または~PII，それらから導出されるもの ］を~webに公開する場合、それによる明白な影響iを軽減する仕方を考慮することが重要になる。 一例として： ◎ If the specification under consideration exposes personal information or PII or their derivatives that could still identify an individual to the web, it’s important to consider ways to mitigate the obvious impacts. For instance:

  • 生体計量~data（指紋や網膜の走査）を利用する特能は、生~dataを~webに公開するのを拒否するベキである。 生~dataの利用は，何らかの［ 生成元に特有，かつ短命な秘匿情報 ］を~unlockすることに限った上で、その秘匿情報を代わりに伝送すること。 ◎ A feature which uses biometric data (fingerprints or retina scans) should refuse to expose the raw data to the web, instead using the raw data only to unlock some origin-specific and ephemeral secret and transmitting that secret instead.
  • 利用者が明示的に選ぶこと（および、希望的には，利用者が解すること）を経ない限り，~dataは公開されないことを確保するため、利用者の仲介を要因に含めることを考慮するベキである。 これを達成し得る仕方には、次が挙げられる ⇒＃ `Permission API^cite `PERMISSIONS$r の利用／ `Payment Request API^cite `PAYMENT-REQUEST-API$r における追加的な~dialog ◎ Including a factor of user mediation should be considered, in order to ensure that no data is exposed without a user’s explicit choice (and hopefully understanding). One way to achieve this may be the use of Permission API [PERMISSIONS], or additional dialogs like in Payment Request API [PAYMENT-REQUEST-API]

2.4. この仕様は、敏感な情報をどう処するか？

~dataが［ 個人的~情報／個人識別可能な情報 ］でないからといって、敏感な情報でないことにはならない。 加えて，所与の情報が敏感かどうかは、利用者ごとに変わり得る。 敏感かどうか考慮する~dataには、［ 財務~data, 資格証, 健康~情報, 所在 ］などが含まれる。 そのような~dataが~webに公開されるときは、公開することによる~riskを軽減する手続きが踏まれるベキである — 例えば： ◎ Just because data is not personally information or PII, that does not mean that it is not sensitive information; moreover, whether any given information is sensitive may vary from user to user. Data to consider if sensitive includes: financial data, credentials, health information, location, or credentials. When this data is exposed to the web, steps should be taken to mitigate the risk of exposing it; for example:

  • `Credential Management^cite `CREDENTIAL-MANAGEMENT$r は、利用者が素早く かつ容易に署名するために，利用者の資格証を~UAの~password~managerに要請することを~siteに許容する。 これは、濫用の扉を開く — ~XSS脆弱性が一つでもあれば、利用者~dataを自明に~JSに公開することになり得る。 ◎ Credential Management [CREDENTIAL-MANAGEMENT] allows sites to request a user’s credentials from a user agent’s password manager in order to sign the user in quickly and easily. This opens the door for abuse, as a single XSS vulnerability could expose user data trivially to JavaScript.＼

    `Credential Management^cite ~APIは、次により~riskを軽減している： ◎ The Credential Management API mitigates the risk by＼

    • ［ ~username, ~password ］を［ ~JSからは直に読取れない不透明な `FormData$I ~obj ］としてのみ提供している。 ◎ offering the username and password as only an opaque FormData object which cannot be directly read by JavaScript and＼
    • 流出の~riskを更に軽減するため、［ 適理な `connect-src$dir, `form-action$dir 値を伴わせた~CSP `CSP$r ］を利用するよう，作者に強く示唆している。 ◎ strongly suggests that authors use Content Security Policy [CSP] with reasonable connect-src and form-action values to further mitigate the risk of exfiltration.
  • 地理所在~情報は、~UAが提供できるより ずっと粗い精度で，多くの利用事例を~serveできる。 一例として，料理店の推奨は、利用者の所在を正確aな~centimeterの位置でなく，市区~levelで依頼することで生成できる。 ◎ Geolocation information can serve many use cases at a much less granular precision than the user agent can offer. For instance, a restaurant recommendation can be generated by asking for a user’s city-level location rather than a position accurate to the centimeter.
  • `GEOFENCING$r 提案は、自身を~swに — したがって、暗号化され, 認証された生成元に — 束ねる。 ◎ A Geofencing proposal [GEOFENCING] ties itself to service workers and therefore to encrypted and authenticated origins.

2.5. この仕様は、複数の閲覧~sessionにわたって持続する新たな状態を生成元~用に導入するか？

複数の閲覧~sessionにわたって利用者の機器~上に~dataを持続することを生成元に許容することは、利用者の知らない所で利用者の制御なしに，この状態が利用者を追跡するために利用され得る~riskを — 当事者主体, 第三者主体どちらの文脈にも — 導入し得る。 状態の持続性を得る新たな仕組みは、［ 利用者を複数~domainにわたって追跡するために利用される／ この状態を消去する制御を伴わずに利用される ］ことを防止する軽減策なしには，導入されるベキでない。 加えて、~UAが特別に考慮するベキ特有な~cacheはあるか？ ◎ Allowing an origin to persist data on a user’s device across browsing sessions introduces the risk that this state may be used to track a user without their knowledge or control, either in a first party or third party contexts. New state persistence mechanisms should not be introduced without mitigations to prevent it from being used to track users across domains or without control over clearing this state. And, are there specific caches that a user agent should specially consider?

例えば： ◎ For example:

  • ~sw `SERVICE-WORKERS$r は、［ ある生成元が為す要請~すべてを横取りすることで、~offlineのときでも完璧に機能する ］ことを，~siteに許容する。 しかしながら，悪意的に注入された~swは、破滅的になる（その仕様の § ~security考慮点 に文書化されたように）。 その仕様では、［ ~swを登録するためには、暗号化され, 認証された接続を要求する ］ことにより，［ `能動的~network攻撃者$／`~XSS攻撃$ ］に対する脆弱性による~riskを軽減している。 ◎ Service Worker [SERVICE-WORKERS] intercept all requests made by an origin, allowing sites to function perfectly even when offline. A maliciously-injected service worker, however, would be devastating (as documented in that spec’s security considerations section). They mitigate the risks an active network attacker or XSS vulnerability present by requiring an encrypted and authenticated connection in order to register a service worker.
  • ~platformに特有な~DRM実装は、［ 利用者を識別する／ ~mediaを成す特定の一片への~accessは是認されるべきかどうか決定する ］ことを助けるため，生成元に特有な情報を公開するかもしれない。 これらの種類の識別子は、濫用をどう軽減できるか決定するため，注意深く評価されるベキである — 利用者が容易に変更し得ない識別子は、追跡の視点からは とても高価値であり，`能動的~network攻撃者$から識別子を保護することが重要な懸念になる。 ◎ Platform-specific DRM implementations might expose origin-specific information in order to help identify users and determine whether they ought to be granted access to a specific piece of media. These kinds of identifiers should be carefully evaluated to determine how abuse can be mitigated; identifiers which a user cannot easily change are very valuable from a tracking perspective, and protecting the identifiers from an active network attacker is an important concern.
  • ~HTTPの［ ~cookie, `ETag^h, `Last-Modified^h ］, `Local Storage^cite, `Indexed DB^cite, 等々。 これらはどれも、利用者についての情報を格納して，後で それを直接的または間接的に検索取得することを生成元に許容する。 ~UAは、［ これらの種類の格納の仕組みが，持続的な識別子を形成することになる~risk ］を［ そのように格納された~dataを一掃する能を，利用者に提供する ］ことにより軽減する。 ◎ Cookies, ETag, Last Modified, Local Storage, Indexed DB, etc. all allow an origin to store information about a user, and retrieve it later, directly or indirectly. User agents mitigate the risk that these kinds of storage mechanisms will form a persistent identifier by offering users the ability to wipe out the data contained in these types of storage.

2.6. この仕様により生成元に公開される下層の~platformからの情報は何か？

そうならば，下層の~platformから公開される情報（例：環境設定~data）は、各 生成元にわたって，一貫するか？ これらには、少なくとも［ 利用者の環境設定, 各種~sensorを含む~system情報, 通信~手法 ］に関係している情報が含まれる。 ◎ If so, is the information exposed from the underlying platform consistent across origins? This includes but is not limited to information relating to the user configuration, system information including sensors, and communication methods.

ある仕様が~host【~host機器】に特有な情報を生成元に公開していて、 その情報が，［ まれにしか変更されない, かつ 各 生成元にわたって変動し得ない ］場合、それは， 2 つの生成元にわたって利用者を一意に識別するために利用し得る — 情報を成す一片が一意であることから直に，あるいは［ 情報を成す異種の各片の組合n ］を利用して一意な指紋を形成できることから `DOTY-FINGERPRINTING$r 。 仕様, および~UAは、［ 可用な情報がどう表面化されるか／ ~software, ~hardware両者における層の重ね方（ `stack^en ）の相違点 ］を注意深く考慮して，指紋収集の~riskを扱うベキである。 指紋収集-能を抑制することは、一貫性を確保すること — 例：可用な~fontたちが成す~listの順序付け — と~~同程度に単純なこともあれば，もっと複階的になることもある。 ◎ When a specification exposes specific information about a host to an origin, if that information changes rarely and is not variable across origins, then it can be used to uniquely identify a user across two origins — either directly because any given piece of information is unique or because the combination of disparate pieces of information are unique and can be used to form a fingerprint [DOTY-FINGERPRINTING]. Specifications and user agents should treat the risk of fingerprinting by carefully considering the surface of available information, and the relative differences between software and hardware stacks. Sometimes reducing fingerprintability may as simple as ensuring consistency, i.e. ordering the list of fonts, but sometimes may be more complex.

そのような情報は、［ 【利用者が】一意に識別される／期待に反して~dataが流出-可能になる ］のを防止するため，生成元に露呈されるベキでない — 仕様における軽減策が，利用者の知る所で利用者の同意tにより~~制止された場合は別として。 ◎ Such information should not be revealed to an origin without a user’s knowledge and consent barring mitigations in the specification to prevent the information from being uniquely identifying or able to unexpectedly exfiltrate data.

例えば： ◎ For example:

  • 一部の~WebGL実装は、文字列 `GL_RENDERER^l を公開し，ある種類の~appにおいて処理能を改善するが、利用者の指紋に持続的な状態を追加する~costと引き換えになる。 この種の機器~levelの詳細には、注意深く~~優先度を与えて，~costより便益が勝ることを確保するベキである。 ◎ The GL_RENDERER string exposed by some WebGL implementations improves performance in some kinds of applications, but does so at the cost of adding persistent state to a user’s fingerprint. These kinds of device-level details should be carefully weighed to ensure that the costs are outweighed by the benefits.
  • ~DOMを介して公開される `NavigatorPlugins$I ~listは、実施上は，ほとんどの利用者において決して変化しない。 一部の~UAは、 ~plugin~listの直な列挙を許容しないことにより，導入される~entropyを抑制する手続きを踏んでいる。 【関係する仕様】 ◎ The NavigatorPlugins list exposed via the DOM practically never changes for most users. Some user agents have taken steps to reduce the entropy introduced by disallowing direct enumeration of the plugin list.

2.7. この仕様は、利用者の機器~上の~sensorに対する~accessを生成元に許容するか？

そうならば，この標準は、どんな種類の［ ~sensor, 情報 ］を［ それらの~sensorから導出して，生成元に公開する ］か？ ◎ If so, what kind of sensors and information derived from those sensors does this standard expose to origins?

~sensorからの情報は、各 生成元にわたる指紋収集~路として~serveし得る。 加えて，~sensorは、利用者の機器や環境について何かを露呈し，その事実は敏感とされる何かになるかもしれない。 加えて，仕様が書かれた頃に その場にあった軽減策は、技術が進歩して脅威~~状況が変化したときには，考慮し直す必要が生じ得る。 ◎ Information from sensors may serve as a fingerprinting vector across origins. 3.1. 受動的~network攻撃者

`受動的~network攻撃者@ は、通信している利用者と~serverの間で伝送路を行き交う~bit列に対する読取n~accessを有する。 攻撃者は、その~bit列を`改変できない^emが，収集して分析できる。 ◎ A passive network attacker has read-access to the bits going over the wire between users and the servers they’re communicating with. 3.3. 同一-生成元~施策に対する違反

`同一-生成元~施策@ は、~webにおける~securityの礎石である — 生成元は、別の生成元の~dataへ直な~accessを有するベキでない（この施策は、 `RFC6454$r の§ 3 にもっと正式に定義される）。 この施策の系として、生成元は `どの生成元にも^em 結付けられていない~data — 一例として，利用者の~hard-driveの内容 — に対しても，直な~accessを有するベキでない。 様々な種類の攻撃が、どうにかして，この保護を迂回しようとする。 例えば： ◎ The same-origin policy is the cornerstone of security on the web; one origin should not have direct access to another origin’s data (the policy is more formally defined in Section 3 of [RFC6454]).   • `~XSS攻撃@ （ `cross-site scripting^en ）は、~target生成元の文脈の中で，攻撃者が制御する~codeを実行させようと騙している攻撃者を孕む。 ◎ Cross-site scripting attacks involve an attacker tricking an origin into executing attacker-controlled code in the context of a target origin.

  • ~data漏洩は、情報を成す~bit列が，不用意に非同一-生成元に可用にされたとき生じる — ~CORS~header `CORS$r を介して明示的に, あるいは `TIMING$r の様な~side-channel攻撃を介して暗黙的に。 ◎ Data leakage occurs when bits of information are inadvertently made available cross-origin, either explicitly via CORS headers [CORS], or implicitly, via side-channel attacks like [TIMING].

3.4. 第三者主体による追跡

~webの力の一部を成すものには、［ 内容／~siteにおける利用者~体験 ］を増補するためとして，~pageが他の第三者主体から内容（画像から~JSまでに至る）を~pullする能がある。 しかしながら，第三者主体から内容を~pullすることは、［ 利用者を追跡して~profileするために利用され得る何らかの情報（~referer, その他の情報など） ］を第三者主体へ漏洩することを，内来的に伴う。 これには，初期~時に格納された~cookieが~domainへ還って行く事も含まれ、非同一-生成元による追跡を許容している。 さらには，第三者主体は、~web~pageが内包する第三者主体~JSを通して，実行~力を~~獲得できる。 ［ ~pageは、第三者主体の内容による~riskを軽減する手続きを踏める ］としても，［ ~browserは、所与の~pageを成す内容が［ 当事者主体, 第三者主体 ］のどちらに属するかに応じて，扱いを違え得る ］としても、新たな機能性が（当事者主体の~siteでなく）第三者主体により実行される~riskは，開発~過程において考慮されるベキである。 ◎ Part of the power of the web is its ability for a page to pull in content from other third parties — from images to javascript — to enhance the content and/or a user’s experience of the site.   • 最も単純な例は、［ 特定の条件~下では，挙動が他と異なる~link ］を~siteに注入することである — 例えば、利用者が~siteに~log-inしたかどうかの事実に基づいて。 これは、利用者が その~siteに~accountを有するかどうかを露呈し得る。 ◎ The simplest example is injecting a link to a site that behaves differently under specific condition, for example based on the fact that user is or is not logged to the site. 4. 軽減~策

この質問票を埋めていくに伴い，当の仕様~内で識別された［ ~security, ~privacy ］の~riskを軽減するためとして、以下に述べる軽減策のいくつかを，当の特能に適用することもできる。 ◎ To mitigate the security and privacy risks you’ve identified in your specification as you’ve filled out the questionnaire, you may want to apply one or more of the mitigations described below to your feature.

4.1. ~data最小化

最小化は、［ 他の通信~相手に公開する情報 ］を［ 所与の演算を完了するために要求される分に限るよう抑える ］ことを孕む策である。 より特定的には、次のいずれかを要求する： ◎ Minimization is a strategy that involves exposing as little information to other communication partners as is required for a given operation to complete. More specifically, it requires＼

  • 利用者が~accessを仲介する際に~~明らかになったものを超える情報へは，~accessを供さない。 ◎ not providing access to more information than was apparent in the user-mediated access or＼
  • 正確にどの情報が供されるかについて，何らかの制御を利用者に許容する。 ◎ allowing the user some control over which information exactly is provided.

例えば，利用者が所与の~fileへの~accessを供した場合、［ それを表現している~objから、~fileの親~directoryやその内容についての情報を得ること ］は，アリにされるベキでない — それらは、明瞭に期待されるものではないので。 ◎ For example, if the user has provided access to a given file, the object representing that should not make it possible to obtain information about that file’s parent directory and its contents as that is clearly not what is expected.

~data最小化の文脈の下では、自ずと，次を尋ねることになる ⇒＃ 異なる主体の間で渡し回される~dataは何か？／ ~data~itemや識別子はどう持続するか？／ 稼働している異なる~protocolの間で相関される可能性はあるか？ ◎ In context of data minimization it is natural to ask what data is passed around between the different parties, how persistent the data items and identifiers are, and whether there are correlation possibilities between different protocol runs.

例えば， `W3C Device APIs Working Group^cite は、~privacy要件~文書 `DEVICE-API-PRIVACY$r にて，いくつかの要件を定義している。 ◎ For example, the W3C Device APIs Working Group has defined a number of requirements in their Privacy Requirements document. [DEVICE-API-PRIVACY]

~data最小化は、仕様の［ 策定者／実装者 ］にも，末端向けに~serviceを配備している者にも適用-可能である。 ◎ Data minimization is applicable to specification authors and implementers, as well as to those deploying the final service.

例として、~mouse~eventを考える。 ~pageが読込まれるとき，~appには［ ~mouseは装着されているかどうか, ~mouseの種別は何か（例： 製造元や~model）, それは どんな種類の能力を公開するか, いくつ装着されているか, 等々 ］を知る仕方は無い。 これらの情報の一部は、利用者が~mouseを利用するものと — おそらく，ヤリトリに要求されたため — 裁定したときに限り，可用になる。 その場合でも、最小な情報のみが公開される： 一例として，それが~trackpadであるかどうかを知ることはできず、右~buttonを備える事も，利用された場合に限り公開される。 `Gamepad API^cite は、この~dataを最小化~能力に用立てる。 ~Web~gameにとっては［ ~UAは~gamepadへの~accessを有するかどうか, いくつ在るか, どんな能力があるか, 等々 ］を知ることは不可能であり、単純に［ 利用者が~gamepadを通して~gameとヤリトリしたいと望む場合に限り，~gamepadに動作したことが~gameに知れることになる ］こと，および［ 動作した結果，演算するために必要なすべての（かつ，それを超えない）情報が~appに供されることになる ］ものと見做される。 ◎ As an example, consider mouse events. When a page is loaded, the application has no way of knowing whether a mouse is attached, what type of mouse it is (e.g., make and model), what kind of capabilities it exposes, how many are attached, and so on. Only when the user decides to use the mouse — presumably because it is required for interaction — does some of this information become available. And even then, only a minimum of information is exposed: you could not know whether it is a trackpad for instance, and the fact that it may have a right button is only exposed if it is used. For instance, the Gamepad API makes use of this data minimization capability. It is impossible for a Web game to know if the user agent has access to gamepads, how many there are, what their capabilities are, etc. It is simply assumed that if the user wishes to interact with the game through the gamepad then she will know when to action it — and actioning it will provide the application with all the information that it needs to operate (but no more than that).

~mouse用の機能性が~supportされる仕方は、単純に［ 一定の~eventが その場を占めたときに限り，~mouseの挙動についての情報を供する ］ことによる。 したがってこの~approachでは、機器への~interfaceとして，~eventの取扱い（例： `onclick^m, `onmousemove^m, `onmousedown^m, 等々を誘発する）のみを公開する。 ◎ The way in which the functionality is supported for the mouse is simply by only providing information on the mouse’s behaviour when certain events take place. The approach is therefore to expose event handling (e.g., triggering on click, move, button press) as the sole interface to the device.

可用にされる~dataを最小化した特能の例には、次の 2 つが挙げられる： ◎ Two features that have minimized the data they make available are:

  • `BATTERY-STATUS-API$r ⇒ “`~UAは高-精度な読出しを公開するベキでない^em” ◎ [BATTERY-STATUS-API] “The user agent should not expose high precision readouts”
  • `SENSORS-API$r ⇒ “`最大~標本化~frequencyを制限する^em”／ “`正確性を抑制する^em” ◎ [SENSORS-API] “Limit maximum sampling frequency”, “Reduce accuracy”

4.2. 既定の~privacy設定群

利用者は、既定の設定を変更しないことが多い。 結果として，仕様の既定の~modeにおいては、公開される［ ~data／識別子 ］の［ 量, 識別-能, 持続性 ］を最小化することが重要になる。 これは特に、当の~protocolが［ 特定の環境に誂えれるような柔軟な~options ］を伴う場合が該当する。 ◎ Users often do not change defaults, as a result, it is important that the default mode of a specification minimizes the amount, identifiability, and persistence of the data and identifiers exposed. This is particularly true if a protocol comes with flexible options so that it can be tailored to specific environments.

4.3. 明示的な利用者~仲介

仕様において，特能の［ ~security／~privacy ］~riskを他では軽減し得ない場合、アリな軽減として，任意選択で利用者に~promptすることを実装者に許容することが最善になることもある — が，それを【利用者が~promptを？】解することは、~privacy~riskをまるごと除去するものではない。 仕様が~promptすることを実装者に許容しない場合、一部の~UAが，もっと~privacyに親切な~versionを実装することを選ぶ結果、各~UAで実装が分岐する結果になりかねない。 ◎ If the security or privacy risk of a feature cannot otherwise be mitigated in a specification, optionally allowing an implementer to prompt a user may be the best mitigation possible, understanding it does not entirely remove the privacy risk. If the specification does not allow for the implementer to prompt, it may result in divergence implementations by different user agents as some user agents choose to implement more privacy-friendly version.

特能の~riskは、それ自体が特能の成り立ちである場合，軽減し得ないこともある。 一例として， `GEOLOCATION-API$r は、利用者の所在を意図的に露呈する。 一般に~UAは、受容するかどうか利用者が選べる許可~prompt上で，当の特能への~access可否を~~制御する。 この~riskは、個人的~dataや識別子を公開する特能にも在るので，それらにも織り込まれるベキである。 ◎ It is possible that the risk of a feature cannot be mitigated because the risk is endemic to the feature itself. For instance, [GEOLOCATION-API] reveals a user’s location intentionally; user agents generally gate access to the feature on a permission prompt which the user may choose to accept. This risk is also present and should be accounted for in features that expose personal data or identifiers.

そのような~promptを設計する困難さは、供するベキ許可の持続時間を決定することにある。 ◎ Designing such prompts is difficult as is determining the duration that the permission should provide.

最良な~promptは、~file~pickerの様に，利用者の動作に明瞭に束ねられるものが多い — そこでは、利用者の動作に呼応して，~file~pickerが持込まれ，利用者は特定の~fileへの~accessを個々の~siteに与える。 ◎ Often, the best prompt is one that is clearly tied to a user action, like the file picker, where in response to a user action, the file picker is brought up and a user gives access to a specific file to an individual site.

一般的に言えば，~prompt【による許可】の持続時間と時機は、~dataを公開することによる~risk【の大きさ】に反比例するベキである。 加えて，~promptは、次に挙げる課題を考慮するベキである： ◎ Generally speaking, the duration and timing of the prompt should be inversely proportional to the risk posed by the data exposed. In addition, the prompt should consider issues such as:

4.5. ~secure文脈

当の仕様~内で，ある特能にて識別された首な~riskが`能動的~network攻撃者$による脅威である場合、~secureでない生成元に特能を提供することは，その特能をどの生成元にも提供するのと同じことになる — 攻撃者は~frameや~codeを意のままに注入できるので。 この種類の~riskは、［ 特能を利用するためには，暗号化され, 認証された接続を要求する ］ことで軽減できる。 ◎ If the primary risk that you’ve identified in your specification is the threat posed by active network attacker, offering a feature to an insecure origin is the same as offering that feature to every origin because the attacker can inject frames and code at will. Requiring an encrypted and authenticated connection in order to use a feature can mitigate this kind of risk.

~secure文脈はまた、`受動的~network攻撃者$に抗して保護する。 例えば、ある~pageが `Geolocation API^cite を利用していて，~sensorから供された［ 緯度, 経度 ］を~secureでない接続~越しに~serverへ送信した場合、どの`受動的~network攻撃者$も 利用者の所在を学習でき，利用者~その他にとって実現可能な検出-法は無い。 ◎ Secure contexts also protect against passive network attackers. しかしながら，~secure文脈を要求するだけでは、`能動的~network攻撃者$ 以外の脅威~動作者からの 多くの~privacy~risk, あるいは~security~riskでさえも，軽減するには足らない。 ◎ However, requiring a secure context is not sufficient to mitigate many privacy risks or even security risks from other threat actors than active network attackers.

4.6. 特能を落とす
