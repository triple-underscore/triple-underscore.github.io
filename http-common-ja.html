Hypertext Transfer Protocol（ HTTP ）

~HTTPを定義する各種~文書

~HTTPは、分散型の協調的な~hypertext情報~system用の，~statelessな応用~levelの~protocolである。 ◎ The Hypertext Transfer Protocol (HTTP) is a stateless application-level protocol for distributed, collaborative, hypertext information systems.

~HTTPは、以下に挙げる［ ~IETF ~RFC, IANA ~registry ］により定義される。 ◎ HTTP is defined by these IETF RFCs and IANA registries.

中核~仕様

~HTTP~protocolの “中核” を成す意味論は、次の仕様により定義される： ◎ The “core” semantics of the HTTP protocol are defined by this specification:

  • RFC7231： ~HTTPの意味論と内容 【意味論は、 RFC 7230 の一部, 主に `RFC7230/2$sec においても定義される。】 ◎ RFC7231: HTTP Semantics and Content

そのような意味論を “伝送路~上” に表出する仕方には、 ~HTTP11, ~HTTP2 の 2 つがある： ◎ Those semantics are expressed “on the wire” in two different ways:

  • RFC7230： ~HTTP11 ◎ RFC7230: HTTP/1.1
  • RFC7540： ~HTTP2（外部~site） ◎ RFC7540: HTTP/2
  • RFC8740： HTTP/2 with TLS 1.3 ◎ RFC8740: HTTP/2 with TLS 1.3

~HTTP2は、次も利用する： ◎ HTTP/2 also uses:

  • RFC7541： HPACK: ~HTTP2用の~header圧縮（外部~site） ◎ RFC7541: HPACK Header Compression for HTTP/2

次に挙げる~HTTP機能を実装するのは任意選択~であるが、広範に利用されている： ◎ These HTTP functions are optional to implement, but widely used:

  • RFC7232： ~HTTP条件付き要請 ◎ RFC7232: HTTP Conditional Requests
  • RFC7233： ~HTTP範囲~要請 ◎ RFC7233: HTTP Range Requests
  • RFC7234： ~HTTP~caching ◎ RFC7234: HTTP Caching
  • RFC7235： ~HTTP認証 ◎ RFC7235: HTTP Authentication

これらの~RFCは、総集的に，~HTTPを定義している先行する~RFCすべてを — RFC1945, RFC2068, RFC2616, RFC2617 を含め — 廃用にする。 ◎ These RFCs collectively obsolete all preceding RFCs defining HTTP, including RFC1945, RFC2068, RFC2616 and RFC2617.

関連する仕様として，数を増やしていく［ `~header$／`~trailer$ ］に利用される次もある ⇒ RFC8941： ~HTTP用の有構造~field値 ◎ Also relevant is the Structured Field Values specification, which is used in a growing number of headers and trailers: • RFC8941: Structured Field Values for HTTP

構文の表記法

この仕様は、 `RFC5234$r による~ABNF（ `Augmented Backus-Naur Form^en ）表記法を利用する。 ◎ This specification uses the Augmented Backus-Naur Form (ABNF) notation of [RFC5234],＼

加えて，この表記法は、 `RFC7405$r に定義される［ 文字列による表記法, その文字大小が区別されるかどうかを指示する表記法 ］で拡張して利用される。 ◎ extended with the notation for case-sensitivity in strings defined in [RFC7405].

【 `%s^c が接頭されていれば文字大小は区別され, `%i^c が接頭されていれば区別されない。 この拡張を利用する仕様は、一部の~RFC／~Internet-Draftに限られる（特に `RFC7405^r 以前のものには利用されない）。 】

また、 `RFC7230/7$sec（改訂~下にある~HTTPにおいては `Semantics/5.6.1$sec ）にて定義される，~list拡張を利用する — それは、（ "`*^c" 演算子が指示する繰返しに類似な）~list演算子（ "`#^c" ）を利用して，~commaで分離された~listを簡潔に定義できるようにする。 ◎ with a list extension, defined in Section 7 of [RFC7230], that allows for compact definition of comma-separated lists using a '#' operator (similar to how the '*' operator indicates repetition).

規約として、［ 接頭辞に "`obs-^p" が付与された~ABNF規則~名 ］が［ 歴史的な理由で出現する “廃用にされた” 文法~規則 ］を記すために用いられる。 ◎ As a convention, ABNF rule names prefixed with "obs-" denote "obsolete" grammar rules that appear for historical reasons.

この仕様にて利用される用語： “文字”, “文字~符号化~scheme”, “~charset”, “`~protocol要素@†” は、`RFC6365$rにて定義される。 ◎ This specification uses the terms "character", "character encoding scheme", "charset", and "protocol element" as they are defined in [RFC6365].

日本語訳に特有な表記規約

~HTTPの日本語訳では、次に挙げる~styleが共通して利用される：

  • `~field名$： `Example-Field^h
  • `要請~method$： `EXAMPLE-METHOD^m
  • `応答~状態s~code$（および，その`事由~句$）： `123^st0 (Example Phrase)
  • `~cache制御~指令$： `example-directive^dir
  • `~protocol要素$（~ABNFの生成規則）： `example-protocol-element^p
  • 他の~code類： `example_keyword^c, "`example_literal^c" （`~protocol要素$の定数~値や~keyword, HTTP ~messageを構成する他の~codeなど）
  • 定義される用語： `用語~例^dfn
  • 変数： `変数例^V — 同じ~styleは、同一の~instance（ “個体” ）を指していることを明示するためにも用いられる： 例えば、複数の文に渡る語 “応答” が同じ応答の~instanceを指しているとき、（例えば他の応答と区別するために） `応答^V と記して，そのことを明示する（それらの文を互いに関連付ける）。

その他、個別の仕様ごとに，特有な~styleが利用されることもある。

日本語への翻訳に関する注記

RFC 原文は，ほぼ素なテキストであり、この日本語訳では、 HTML マークアップ ／ CSS スタイル の利用により記述が等価になる所では、一部の対訳を省略している。 例えば：

  • 用語の定義箇所を示す，節 番号による参照の多くは、 代わりに用語そのものにリンクをあてがった上で，節 番号の記述を省略している。
  • リンク先の説明で代用になるときには、 記述を省くことがある。
  • 一部の語 — 例： “状態°コード（ status code ）” — は、 コード番号が記されている所では， （その番号に専用のスタイルがあてがわれるので）自明なので省略することもある。

また、見通しを良くするため、原文のかなりの割合の段落は， マークアップを利用して より明示的な文脈階層に細分化している 。

一部の語の対訳は，［ “忠実さに欠く” 語／一般に見受けられる語と異なる語 ］に代えている — 例えば：

  • 原文の語 “media type” （媒体型, Internet media type の略称）は、 RFC 以外の Web 標準に共通して利用されている用語 “MIME 型” に一律に置換している。
  • “回送（ forward ）” ： “転送” あるいは “自動転送” という対訳もあるが、 前者は “transfer” の対訳と衝突し， 後者は長い（また， “automatic な transfer” と誤解され易くもなる）ので。
  • “条態（ condition ）” ： これは、訳者による造語（条件＋状態）である。 “状況／状態” , あるいは “条件” と訳されることが多いが、前者は “situation”, “circumstance” ／ “state” などの対訳と衝突し，後者は文脈にそぐわないことが多々あるので（無論、 “コンディション” とも訳せるが，用語 “条件付き（ conditional ）” との関連性もあるので）。
  • “全部的（ full ）” — “完全（ complete ）” との区別を要するため。 また、 “部分的（ partial ）” との対比も兼ねるため。
  • “検証°（ verify ）” （正しく作られたかどうかを検証） — “検証（ validate ）” （作られたものが正しいかどうかを検証）と区別するため。
  • “状態°（ status ）” （状態の集合／状態を［指示する／識別する］もの） — “状態（ state ）” （ある特定の状態）と区別するため。
